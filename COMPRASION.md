# Сравнение `APIView` vs `ModelViewSet` в Django REST Framework

При проектировании эндпоинтов в DRF выбор уровня абстракции влияет на скорость разработки, читаемость и поддержку кода.

Ниже — подробное сравнение низкоуровневого `APIView` и высокоурневого `ModelViewSet`.

---

# 1. `APIView` — низкоуровневый подход

`APIView` — это базовый класс DRF, максимально близкий к обычным Django Views.

Разработчик вручную связывает логику с HTTP-методами:

- `get`
- `post`
- `put`
- `patch`
- `delete`

---

## Плюсы

### Полный контроль

Вы полностью управляете обработкой запроса:

- как получать данные из БД
- как валидировать данные
- как формировать ответ
- какие ошибки возвращать

---

### Гибкость для не-CRUD логики

Подходит для эндпоинтов, которые не связаны напрямую с моделью.

Примеры:

- отправка SMS-кодов
- авторизация
- интеграция со сторонними API
- генерация отчетов
- сложные вычисления

---

### Явность кода

Открывая метод `get()` или `post()`, сразу видно всю логику.

Меньше скрытой «магии» DRF.

---

## Минусы

### Много шаблонного кода (Boilerplate)

Для каждой сущности приходится вручную писать:

- получение объекта
- обработку `404`
- валидацию сериализатора
- сохранение данных
- формирование ответа

---

### Ручная маршрутизация

Все URL нужно прописывать вручную в `urls.py`.

Пример:

```python
path("tasks/", TaskListAPIView.as_view()),
path("tasks/<int:pk>/", TaskDetailAPIView.as_view()),
```

---

# 2. `ModelViewSet + Router` — высокоуровневый подход

`ModelViewSet` объединяет всю CRUD-логику для модели в одном классе.

Вместо HTTP-методов используются actions:

- `list`
- `create`
- `retrieve`
- `update`
- `partial_update`
- `destroy`

---

## Плюсы

### Максимальная скорость разработки

Достаточно указать:

```python
queryset = Task.objects.all()
serializer_class = TaskSerializer
```

и DRF автоматически создаст CRUD API.

---

### Автоматическая маршрутизация через Router

`DefaultRouter` сам генерирует URL.

Пример:

```python
router.register("tasks", TaskViewSet)
```

Создаются маршруты:

```text
GET     /tasks/
POST    /tasks/
GET     /tasks/1/
PUT     /tasks/1/
PATCH   /tasks/1/
DELETE  /tasks/1/
```

---

### DRY (Don't Repeat Yourself)

Логика для сущности хранится в одном месте.

Например:

```python
def get_queryset(self):
    return Task.objects.filter(user=self.request.user)
```

Изменение автоматически применяется ко всем actions.

---

### Кастомные actions

Через `@action` можно добавлять собственные эндпоинты.

Пример:

```python
@action(detail=True, methods=["post"])
def share(self, request, pk=None):
    ...
```

URL:

```text
/api/tasks/1/share/
```

---

## Минусы

### Много скрытой магии

Большая часть логики DRF скрыта внутри:

- mixins
- generic views
- routers

Новичкам бывает сложно понять:

- когда вызывается сериализатор
- где происходит `save()`
- как формируется response

---

### Сложнее для нестандартной логики

Если `POST` должен выполнять полностью кастомную операцию, переопределение `create()` внутри `ViewSet` может выглядеть неудобно.

---

# Сводная таблица

| Критерий | `APIView` | `ModelViewSet` |
|---|---|---|
| Уровень абстракции | Низкий | Высокий |
| Основное назначение | Кастомная логика | CRUD над моделями |
| Маршрутизация | Ручная | Автоматическая через Router |
| Объем кода | Большой | Минимальный |
| DRY | Часто нарушается | Хорошо соблюдается |
| Контроль над логикой | Максимальный | Ограниченный |
| Скорость разработки | Ниже | Очень высокая |
| Простота понимания | Выше | Ниже из-за магии DRF |
| Расширяемость | Гибкая | Удобна для CRUD |

---

# Когда использовать `APIView`

Используйте `APIView`, если:

- нужна сложная кастомная логика
- endpoint не связан с моделью
- важен полный контроль над обработкой запроса
- требуется нестандартная архитектура API

---

# Когда использовать `ModelViewSet`

Используйте `ModelViewSet`, если:

- нужен обычный CRUD
- важна скорость разработки
- проект содержит много типовых сущностей
- хотите минимизировать количество кода

---

# Практическое правило

Во многих проектах используют оба подхода одновременно:

- `ModelViewSet` → для стандартного CRUD
- `APIView` → для сложной бизнес-логики и интеграций

Это считается нормальной и правильной практикой.
